# Plan: create-amytis CLI (Model A)

## Context

Users want to start a new Amytis-based site without forking/cloning the repo manually.
The CLI (`bunx create-amytis my-blog`) will scaffold a full Amytis project by downloading
the latest GitHub release tarball, then asking two setup questions (site title, description)
before patching `site.config.ts` and running `bun install`.

Model A: user gets the full source — `src/`, `scripts/`, `content/`, `site.config.ts`.
They own all the code; only `content/` and `site.config.ts` need to be touched day-to-day.

---

## Package structure

```
packages/
  create-amytis/
    package.json      ← name: "create-amytis", bin: "dist/index.js"
    src/
      index.ts        ← CLI entry point
    dist/             ← compiled output (gitignored, published to npm)
    tsconfig.json
    .gitignore
```

Root `package.json` gets a bun workspace entry:
```json
"workspaces": ["packages/*"]
```

---

## CLI flow

```
bunx create-amytis [project-name]
```

1. Read `project-name` from `process.argv[2]`; if missing, prompt for it
2. Call GitHub API to get latest release tag:
   `GET https://api.github.com/repos/hutusi/amytis/releases/latest`
3. Download tarball:
   `https://github.com/hutusi/amytis/archive/refs/tags/{tag}.tar.gz`
4. Extract into `./{project-name}/`  (tarball root dir is `amytis-{version}/`, rename it)
5. Prompt for **Site title** (default: project-name)
6. Prompt for **Site description** (default: "My digital garden")
7. Patch `site.config.ts`:
   - Replace `title: { en: "Amytis", zh: "Amytis" }` with user title
   - Replace `description: { en: "...", zh: "..." }` with user description
8. Patch `package.json` in scaffolded dir:
   - Set `"name"` to project-name
   - Set `"private": true` (they're not publishing this)
   - Remove `"repository"`, `"bugs"`, `"homepage"` fields
9. Run `bun install` in the new directory
10. Print success + next steps

## Critical files to create

### `packages/create-amytis/package.json`
```json
{
  "name": "create-amytis",
  "version": "0.1.0",
  "description": "Create a new Amytis digital garden",
  "license": "MIT",
  "bin": { "create-amytis": "./dist/index.js" },
  "scripts": {
    "build": "bun build src/index.ts --outfile dist/index.js --target node --sourcemap=none",
    "dev": "bun src/index.ts"
  },
  "files": ["dist"]
}
```

### `packages/create-amytis/src/index.ts`

Key implementation details:
- `#!/usr/bin/env node` shebang
- Zero npm dependencies — use Node.js built-ins only:
  - `https` for API + tarball download
  - `zlib` (gunzip) + `tar` via child_process (`tar xz`) for extraction
  - `readline` for interactive prompts
  - `fs`, `path`, `child_process` for file ops and install
- Helper `prompt(question, defaultVal)` → `Promise<string>` via readline
- Helper `fetchJson(url)` → GitHub API call with `User-Agent` header (required by GitHub API)
- Helper `downloadTarball(url, dest)` → streams tarball to disk
- Helper `extractTarball(tarPath, targetDir)` → runs `tar xzf` via child_process
- Patch functions use `fs.readFileSync` + regex replace + `fs.writeFileSync`

### `packages/create-amytis/tsconfig.json`
Standard Node.js tsconfig (target ES2022, module NodeNext).

## site.config.ts patch strategy

The file has these exact strings to replace:

```ts
title: { en: "Amytis", zh: "Amytis" },
```
→
```ts
title: { en: "My Blog", zh: "My Blog" },
```

```ts
description: { en: "Amytis — an elegant open-source framework...", zh: "Amytis — 优雅的开源数字花园框架。" },
```
→
```ts
description: { en: "My description", zh: "My description" },
```

Use regex with enough context to avoid false matches. The zh value gets the same
user-provided string (user can change it later). The `footerText` field references
"Amytis" by name — also replace those occurrences.
